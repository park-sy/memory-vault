# ADR-002: Platform Team 구조 채택

## Context

현재 구조(Hub-and-Spoke)의 한계:
- Orchestrator가 모든 통신을 중계 → 병목 + 정보 열화
- Worker가 범용 → 전문성이 세션에 축적되지 않음
- 프로젝트별 팀 편성 → 같은 역량 중복 로드, 교차 학습 불가
- 프로젝트 추가 시 팀 전체를 새로 구성해야 함

기존 [[001-vault-structure]]의 옵시디언 vault 구조 위에서, 조직 편제 자체를 재설계하는 결정.

참고: Team Topologies (Matthew Skelton & Manuel Pais)의 Platform Team 모델 기반.
초기에 "헥사고날 아키텍처"로 불렸으나, 실제 패턴은 Platform Team + Stream-aligned Team 구조.

## Decision

### Platform Team 구조

```
┌─────────── Platform Team (코어 전문팀) ───────────┐
│  리서치  |  계획  |  리뷰  |  QA  | ...            │
└──────┬──────────┬──────────┬──────────┬───────────┘
       │          │          │          │
  ┌────┴───┐ ┌───┴────┐ ┌──┴───┐ ┌───┴────┐
  │도메인 A │ │도메인 B │ │도메인C│ │도메인 D │
  │전문가+  │ │전문가+  │ │      │ │        │
  │컨텍스트 │ │컨텍스트 │ │      │ │        │
  └────────┘ └────────┘ └──────┘ └────────┘
       Stream-aligned Team (도메인팀)
```

**Platform Team (코어 전문팀)**: 기능별 전문가 (리서치, 계획, 코드리뷰, QA 등)
- 전문 분야의 지식을 축적 (체크리스트, 패턴, 실수 DB)
- 여러 도메인을 넘나들며 교차 수분
- 필요 시 워커 세션을 추가로 띄워 수평 확장

**Stream-aligned Team (도메인팀)**: 프로젝트별 컨텍스트 + 도메인 전문 개발 담당
- 자체 개발 조직 없이 코어팀을 호출해서 사용
- 도메인 지식과 컨텍스트를 정리/관리
- 강력한 도메인 전문가 개발 담당을 둬서 깊이 보완

**통신**: 모든 팀 간 자유 소통 가능 (Collaboration Mode)

### Hub-and-Spoke 대비 비교

| 차원 | Hub-and-Spoke | Platform Team |
|------|-------------|--------------|
| 토큰 효율 (단순 작업) | 유리 | - |
| 토큰 효율 (협업 작업) | - | 유리 |
| 결과물 퀄리티 | - | 유리 (교차 수분, 일관성) |
| 초기 셋업 | 유리 | 오버헤드 |
| 성장기 확장 | 병목 발생 | 선형 확장 |
| 성숙기 확장 | 파탄 | 자연스러움 |
| 장애 내성 | Orch 죽으면 전체 마비 | 부분 장애 허용 |
| 지식 축적 | 프로젝트별 분산 | 전문 분야별 심화 |

### 잠재적 단점 및 대응

**초기 검토된 단점 (모두 해소됨)**

| 단점               | 대응                              | 해소 수단      |
| ---------------- | ------------------------------- | ---------- |
| 코어팀 병목           | 세션 수평 확장                        | AI 세션 스케일링 |
| 도메인 컨텍스트 반복 로딩   | context.md 정리 + 주입              | 옵시디언 연결    |
| 책임 소재 분산         | 건별 최종 책임자 명시                    | 운영 규칙      |
| 오염 전파            | 도메인 특수 → 도메인 기록 + [[wiki-link]] | 옵시디언 연결    |
| 도메인 깊이 부족        | 도메인팀에 전문가 개발 담당 배치              | 조직 편제      |
| 콜드 스타트           | 코어팀 사전 정의 (패턴/체크리스트 시드)         | 사전 정의      |
| 사소한 작업 오버킬       | LLM 불필요한 작업은 코드로 처리             | 코드 자동화     |
| 멀티팀 핸드오프 유실      | 공유 문서 wiki-link + AI 해석 편차 0    | 옵시디언 + AI  |
| memory 비대화       | dashboard.md의 cold/archive 뷰 활용 | 기존 인프라     |
| context.md 관리 부담 | 프로젝트 구조 스캔 자동 갱신 스크립트           | 코드 자동화     |
| 코어팀 간 패턴 충돌      | wiki-link 그래프 가시화 + 충돌 감지 스크립트  | 옵시디언 + 코드  |

**해소 원칙**: LLM이 필요 없는 영역은 코드로, 팀 간 공유는 옵시디언 구조로, AI 고유 이점(스케일링/해석 동일성/롤백)으로 나머지 해결.

**구현 도구**: [[platform-team-automation]] 참조 — Hook 3개, Script 3개, Skill 3개로 전체 단점 해소.

### AI 에이전트이기 때문에 가능한 이유

사람 조직에서 이 구조가 비효율적인 이유:
- **메모리 공유 불가** → 지식 전달에 커뮤니케이션 비용이 크다
- **Worker 수평 확장 불가** → 전문팀 병목을 해소할 수 없어 각 팀에 전용 인원 배치

AI 에이전트는:

**자원 확장**
- 세션 스케일링 자유 → 역할 "전환"이 아니라 "복제". 필요하면 같은 역할 세션 추가
- 컨텍스트 스위칭 비용 ≈ 파일 로드 (사람은 ~23분)
- 피로/품질 저하 없음 → 100번째 리뷰도 1번째와 동일 품질
- 포킹 가능 → 두 방향 동시 실험, 나은 쪽 채택

**지식 체계**
- 메모리 공유 → 동일 파일 읽으면 동일 지식
- 해석 편차 0 → 10개 세션이 같은 문서 읽으면 동일하게 이해 (사람은 10가지로 해석)
- 암묵지 불가능 → 아는 것 = 파일에 있는 것. "그 사람만 아는 것"이 구조적으로 없음
- 전문성 합성 → 역할 파일 조합으로 즉석 하이브리드 전문가 생성
- 버스 팩터 ∞ → 세션이 죽어도 지식 손실 0
- 타임트래블 → 과거 세션 파일 로드하면 당시 맥락/추론 완벽 복원

**조직 운영**
- 에고/정치 없음 → 팀 간 자유 소통이 진짜로 작동
- 롤백 가능 → memory.md 수정 또는 git revert로 오염 즉시 복구
- git blame → 패턴의 기원과 이유를 코드처럼 추적
- 비용 정량화 → 토큰 = 돈. 조직 구조 효율을 숫자로 측정/최적화 가능

## Consequences

- 코어팀 role.md + memory.md 신규 작성 필요
- 도메인팀 context.md 구조 정의 필요
- 기존 ops/product/engineering 부서 구조를 core/domains로 재편
- 점진적 전환: 코어팀 memory 축적부터 시작
- [[001-vault-structure]]의 디렉토리 구조 일부 변경 필요
