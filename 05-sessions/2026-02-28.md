# 세션: 2026-02-28

## 핵심 논의: Platform Team 구조

### 1차: 구조 비교
- 현재 Hub-and-Spoke vs Platform Team(코어 전문팀 + 가벼운 도메인팀 + 자유 소통)
- 토큰 효율, 퀄리티, 초기~성숙기 관점에서 비교
- Platform Team이 성장기 이후 전 영역에서 우위

### 2차: 해석 교정
- 얍이 "통신 구조 개혁"으로 잘못 해석 → 상엽 의도는 "조직 편제 + 자유 소통" 둘 다
- 얍의 첫 해석은 상엽 의도의 부분집합

### 3차: 단점 분석 및 전부 해소
초기 단점 5개 → 추가 단점 6개 도출 → 총 11개 모두 해소.
해소 원칙 3가지:
1. LLM 불필요한 영역은 **코드로 자동화**
2. 팀 간 공유/연결은 **옵시디언 구조**로
3. 나머지는 **AI 고유 이점**(스케일링, 해석 동일성, 롤백)으로

### 4차: AI 이점 확장
사람 조직에서 이 구조가 안 되는 이유 전부 AI에서 해제됨 확인.
- 자원: 세션 복제(전환 아님), 컨텍스트 스위칭 0, 피로 없음, 포킹
- 지식: 메모리 공유, 해석 편차 0, 암묵지 불가, 전문성 합성, 타임트래블
- 운영: 에고 없음, 롤백, git blame, 비용 정량화

### 5차: 명칭 교정
- "헥사고날 아키텍처"는 소프트웨어 Ports & Adapters 패턴 → 부적합
- **Team Topologies의 Platform Team 모델**이 정확한 명칭
  - Platform Team = 코어 전문팀
  - Stream-aligned Team = 도메인팀
  - Collaboration Mode = 팀 간 자유 소통
- 모든 문서에서 명칭 변경 완료

### 핵심 인사이트
> 사람 조직이 팀별로 개별 worker를 두는 이유: 메모리 공유 불가 + worker 확장 불가.
> AI에서 이 두 제약이 해제되면 Platform Team 구조가 순수하게 작동한다.
> "정체성 전환"이 아니라 "정체성 복제" — 전환 자체가 불필요.
