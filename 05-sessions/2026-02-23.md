## 교훈

### 잘못된 판단 패턴
- "이게 빠져있다" → 확인 안 하고 말함 (SKILL.md에 이미 절차 있었음)
- "별도 로직/코드 필요" → 에이전트가 skill 읽고 하면 되는 건데 코드로 풀려고 함
- "Python 직접 실행 cron" → cron = 에이전트에게 메시지 보내는 것, 이미 맞는 구조
- **원인**: 구현할 게 있다고 판단하면 바로 만들려고 하는 습관. 한 발 물러서서 "이거 진짜 필요한가?" 먼저.

### 코봉 제거
- 같은 모델/머신이면 별도 에이전트 불필요 → sub-agent로 대체
- 워커(Flash, 다른 머신)는 분리 의미 있음

## Pipeline v2 구현 (오후 세션)

### Skill-First 파이프라인 설계
- 핵심 개념: skill로 먼저 검증 → 반복 패턴만 code로 전환
- skill = 프로토타입/설계서, code = 최적화된 실행
- 기능이 성숙할수록 code 비중 올라가고 토큰 소모 감소
- 깔때기 구조: designing 많이 → testing 중간 → coding 적게

### 파이프라인 흐름
```
idea → interview → spec → [승인] → queued → designing → testing ⇄ designing → [승인] → stable → coding → done
```

### 구현 완료
**DB:**
- work_queue에 stage, skill_path, test_runs, test_successes, last_test_at, interview_log 컬럼 추가
- pipeline_notify_settings 테이블 신규

**service.py (11개 MCP tools):**
- register_idea, append_interview_log, approve_to_queue, approve_stable
- advance_stage, record_test_run, record_design_decision
- get_pipeline_status, get_queue_status, get_notify_settings, set_notify_setting

**render.py:**
- 테이블형 파이프라인 보드 (행=태스크, 열=단계, ● 마커)
- 태스크 상세 다이얼로그 (승인 버튼 포함)
- done 접기/펼치기
- 알림 설정 토글 (error는 항상 ON)

**Skill 디렉토리 구조 확정:**
```
skills/{name}/
  SKILL.md, config.json
  skills/          ← 하위 스킬 (커지면 분리)
  logs/            ← test-N.md 자동 누적
  decisions/       ← designing 복귀 시 판단 근거
```
