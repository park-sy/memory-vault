# Planner Skill — Spec 작성 방법론

## 개요

아이디어(idea)를 구체적인 Feature Specification(spec.md)으로 변환하는 방법론.
planner 세션이 사용자와 대화하며 이 방법론을 따라 spec을 작성한다.

## spec.md 필수 6개 섹션

모든 spec은 반드시 아래 6개 섹션을 포함해야 한다.

| # | 섹션 | 핵심 질문 |
|---|------|-----------|
| 1 | **개요** | 이 기능이 무엇이며, 어떤 문제를 해결하는가? |
| 2 | **요구사항** | 사용자/비즈니스 요구사항은 무엇인가? |
| 3 | **기능 상세** | 구체적으로 어떻게 작동하는가? (UI + 백엔드) |
| 4 | **데이터 모델 (DB)** | 어떤 테이블, 컬럼, 관계가 필요한가? |
| 5 | **Execution Levels (LLM Boundary)** | 각 기능이 Level 0/1/2 중 어디인가? |
| 6 | **테스트 계획** | 어떻게 검증할 것인가? |

---

## 3가지 관점 분석 (필수)

### 1. 사용자 관점 (UI/UX)

사용자가 이 기능을 어떻게 경험하는지 정의한다.

- **화면 플로우**: 사용자가 어떤 순서로 어떤 화면을 보는지
- **와이어프레임**: 각 화면의 레이아웃을 **반드시 ASCII 아트로 제시** (텍스트 박스, 테이블, 버튼 등)
- **인터랙션**: 버튼/입력/선택 등 사용자 액션 정의
- **단계 분리**: 한 번에 모든 결과를 뱉지 않고, 사용자가 단계별로 진행

질문 예시:
- "이 기능을 사용할 때 첫 화면에서 뭘 보고 싶어?"
- "결과를 한 번에 다 볼래, 아니면 단계별로?"
- "어떤 버튼/메뉴로 이 기능에 접근해?"
- "데이터가 0건일 때는 어떻게 보여줄까?"
- "이 화면에서 가장 자주 하는 동작이 뭘 것 같아?"

### 2. AI 에이전트 관점 (SKILL 실행)

에이전트가 이 기능을 어떤 흐름으로 실행하는지 정의한다.

- **에이전트 실행 흐름**: CLI command 실행 순서 + 판단 포인트
- **외부 데이터 소스**: 검색, API 등 필요한 외부 연동
- **LLM Boundary 분해**: 각 기능을 Level 0/1/2로 분류

| LLM Boundary | 설명 | 예시 |
|---|---|---|
| Level 0 (코드) | 토큰 0. 코드로 100% 처리 | CRUD, 계산, 파싱, cron |
| Level 1 (정형 LLM) | 템플릿 기반 생성/요약/분류 | 리포트 생성, 텍스트 요약 |
| Level 2 (비정형 LLM) | 에이전트 판단, 복잡한 분석 | 설계 판단, 디버깅 |

질문 예시:
- "이 기능에서 LLM이 꼭 필요한 부분이 어디야?"
- "외부 API나 데이터 소스가 필요해?"
- "에이전트가 판단해야 하는 포인트가 있어?"
- "정확도가 좀 떨어져도 빠르고 싸게 할래, 아니면 느려도 정확하게?"
- "이 기능이 실패하면 fallback이 뭐야?"

### 3. 데이터 관점 (DB 설계)

기능에 필요한 데이터 구조를 정의한다.

- **테이블 구조**: 테이블명, 컬럼, 타입, 제약조건
- **관계도**: 테이블 간 관계 (FK, 참조)
- **CRUD 시나리오**: 어떤 화면에서 어떤 데이터가 생성/조회/수정/삭제되는지
- **migration SQL 초안**: 실행 가능한 CREATE TABLE 문

질문 예시:
- "이 데이터는 어디에 저장돼야 해? DB? 파일?"
- "기존 테이블과 연관 있는 데이터야?"
- "히스토리/로그가 필요해?"
- "이 데이터 얼마나 오래 보관할 거야?"
- "데이터 양이 많아지면 어떻게 관리할까? (페이지네이션, 아카이빙 등)"

## LLM Boundary 분해 방법

모든 기능을 서브 기능으로 쪼갠 뒤, 각각에 Level을 부여한다.

### 분류 기준

```
이 기능을 코드만으로 100% 처리할 수 있는가?
  ├─ YES → Level 0
  └─ NO → LLM이 필요한 이유는?
        ├─ 템플릿 기반 생성/요약/분류 → Level 1
        └─ 복잡한 판단/분석/설계 → Level 2
```

### Execution Levels 테이블 형식

| 기능 단위 | Level | 방식 | 리소스 / 호출 조건 |
|-----------|-------|------|-------------------|
| 데이터 수집 | 0 | 코드 | cron, requests |
| 리포트 생성 | 1 | 프롬프트 | prompts/report.md |
| 오류 판단 | 2 | 에이전트 | error rate > 30% 시 |

## 인터뷰 원칙

1. **애매하면 반드시 물어본다** — 추측하지 않는다. "이게 A인지 B인지 확실치 않은데, 어떤 쪽이야?"
2. **추상적이면 선택지를 제시한다** — "자동으로 해줘"라고 하면 "자동"의 구체적 방식 2~3가지를 비교해서 보여준다.
3. **고정 질문을 기계적으로 밟지 않는다** — 이미 명확한 부분은 스킵하고, 불명확한 부분에 집중한다.
4. **사용자가 놓친 관점을 짚어준다** — "이 부분은 생각해봤어?" 식으로 빈 곳을 능동적으로 발견한다.
5. **결정을 대신하지 않되, 판단 재료를 충분히 준다** — 각 선택지의 장단점, trade-off를 명시한다.

## 사용자와의 대화 패턴

### Phase 1: 이해 (적응형 질문)

아이디어를 받으면 바로 spec을 쓰지 않는다. 먼저 아이디어를 분석하고, **불명확한 부분만 골라서** 질문한다.

1. 아이디어에서 명확한 부분 / 애매한 부분을 내부적으로 분류
2. 명확한 부분은 "~로 이해했는데 맞지?" 식으로 간단히 확인
3. 애매한 부분은 구체적 질문 또는 선택지 제시

**질문 깊이 기준:**

| 아이디어 상태 | 대응 | 예시 |
|---|---|---|
| 구체적 ("DB에 저장하고 리스트로 보여줘") | 바로 Phase 2 | 확인만 하고 넘어감 |
| 방향만 있음 ("여행 기능 만들어줘") | 핵심 2~3개 질문 | "어떤 여행 정보를 다루고 싶어?" |
| 추상적 ("뭔가 생산성 올리는 거") | 선택지 제시 + 깊은 질문 | "생산성이라면 A, B, C 중 어떤 방향?" |

**추상적 표현 → 선택지 제시 패턴:**

사용자가 추상적으로 말하면 반드시 2~3가지 구체적 방안을 비교 테이블로 제시한다.

```
사용자: "AI가 알아서 추천해줬으면 좋겠어"

플래너:
추천 방식을 3가지로 생각해봤어:

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A. 규칙 기반 | 사전 정의된 조건으로 필터링 | 예측 가능, 토큰 0 | 유연성 낮음 |
| B. LLM 정형 | 프롬프트 템플릿으로 추천 생성 | 자연어 품질 높음 | 토큰 소모 |
| C. 하이브리드 | 규칙 필터 → LLM 정렬 | 비용 최적 | 구현 복잡 |

어떤 방향이 맞아?
```

### Phase 2: 구체화 (3가지 관점 — 깊은 탐색)

질문 답변을 바탕으로 3가지 관점을 순서대로 분석한다.

1. 사용자 관점 → 화면 흐름 제안
2. AI 에이전트 관점 → tool 흐름 + LLM Boundary 제안
3. 데이터 관점 → 테이블 구조 제안

각 관점을 설명하고 사용자 피드백을 받는다.

**각 관점에서 추가 탐색할 것:**

- **사용자 관점**: 엣지 케이스 ("데이터가 0건이면?", "입력 중 취소하면?"), 빈도 ("이 기능 얼마나 자주 쓸 것 같아?")
- **AI 관점**: 비용 vs 품질 trade-off ("정확도가 좀 떨어져도 빠르고 싸게 할래, 아니면 정확하게?"), 실패 시 fallback
- **데이터 관점**: 데이터 수명 ("이 데이터 얼마나 오래 보관해?"), 히스토리 필요 여부, 기존 테이블과의 관계

**빈 곳 발견 패턴:**

각 관점 분석 후, 사용자가 언급하지 않았지만 필요한 부분을 능동적으로 짚는다.

```
"한 가지 빠진 게 있는데 — 오프라인 상태에서 이 기능을 쓸 일이 있어?
없으면 넘어가고, 있으면 캐싱 전략을 잡아야 해."
```

### Phase 3: 확인 (spec 초안 → 최종)

1. spec 초안을 한 번에 보여준다
2. **spec 자가 검증**: 초안에서 스스로 약한 부분을 표시 ("이 부분은 좀 더 구체화하면 좋겠는데, 괜찮아?")
3. 사용자 피드백 반영
4. 최종 확인 → `SPEC_READY`

## 좋은 spec vs 나쁜 spec

### 좋은 spec

```markdown
## Execution Levels
| 기능 단위 | Level | 방식 | 리소스 |
|-----------|-------|------|--------|
| 여행지 검색 | 0 | 코드 | Brave Search API |
| 일정 생성 | 1 | 프롬프트 | prompts/itinerary.md |
| 예산 최적화 | 2 | 에이전트 | 사용자 선호도 기반 판단 |

## 데이터 모델
CREATE TABLE trips (
    id INTEGER PRIMARY KEY,
    destination TEXT NOT NULL,
    start_date TEXT NOT NULL,
    end_date TEXT NOT NULL,
    budget_krw INTEGER,
    created_at TEXT DEFAULT (datetime('now'))
);
```

특징:
- Level이 구체적 (왜 Level 1인지 근거가 있음)
- DB 스키마가 실행 가능 (타입, 제약조건 명시)
- 각 기능이 독립적으로 테스트 가능

### 나쁜 spec

```markdown
## 기능 상세
AI가 여행 계획을 자동으로 생성해줍니다.
사용자 입력을 받아서 최적의 일정을 추천합니다.

## DB
trips 테이블 사용
```

문제점:
- "AI가 자동으로" → Level이 뭔지 불명확
- "최적의" → 판단 기준 없음
- DB 스키마가 없음 (컬럼, 타입 불명)
- 테스트 불가능 (기대 결과를 정의할 수 없음)

## 완료 기준

spec이 다음 조건을 모두 만족하면 `SPEC_READY`를 선언한다:

1. 6개 필수 섹션이 모두 존재
2. 3가지 관점(사용자/AI/데이터)이 모두 반영됨
3. LLM Boundary가 기능 단위로 분해됨
4. DB 스키마가 실행 가능한 SQL로 작성됨
5. 테스트 계획이 구체적 (입력 → 기대 출력 형태)
6. 사용자가 최종 확인함

```
SPEC_READY
```
