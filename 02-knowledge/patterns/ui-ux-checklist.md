# UI/UX 구현 체크리스트

factory-monitor 구현 시 상엽의 피드백에서 추출한 원칙. 모든 대시보드/UI 작업에 적용.

## 핵심 원칙 (최우선)

> 코드를 만들 때, 사용자가 충분한 정보를 획득할 수 있고,
> 기능을 세밀하게 조절할 수 있게 만들어라.

**정보 풍부성**: 모든 상태/수치에 맥락을 제공하라.
- 숫자만 보여주지 말고, 그 숫자가 의미하는 바를 함께 보여줘라
- 정상/비정상 기준, 원인, 영향 범위, 다음 행동까지 안내
- "이 값은 뭐고, 지금 괜찮은 건가, 아니면 뭘 해야 하는가"에 답할 수 있어야 함

**세밀한 조절 가능성**: "있다/없다" 수준이 아닌, 실질적 제어를 제공하라.
- 설정이면 min/max뿐 아니라 "이 값을 바꾸면 뭐가 달라지는지" 설명
- 상태면 2단계(on/off)가 아니라 실제 전이 과정을 반영
- 사용자가 시스템을 이해하고 자기 의도대로 조절할 수 있어야 함

이 두 원칙을 만족하지 못하면 아래 개별 항목을 다 지켜도 불충분하다.

## 1. 다크모드 호환

> 색상을 하드코딩하지 마라.

- 배경: `rgba()` 반투명 또는 CSS 변수(`var(--wl-*)`) 사용
- 텍스트: 명시적 색상 지정 금지, 테마 상속에 의존
- **구현 전 다크모드에서 시각적으로 확인할 것**

```python
# BAD
.style("background: #f5f5f5")

# GOOD
.style("background: rgba(128,128,128,0.15)")
```

## 2. 연관 컴포넌트는 묶어라

> 항상 같이 동작하는 것은 하나로 제어해라.

- "A 없이 B가 의미 없다" → 통합 제어 (한 버튼)
- 개별 상태는 인디케이터로 보여주되, 조작은 통합
- 예: 데몬 + 브릿지 → "시스템" 한 버튼, 상태 점으로 개별 표시

```
# BAD: 개별 시작/종료
데몬    [시작] [종료]
브릿지  [시작] [종료]

# GOOD: 통합 제어 + 개별 상태
시스템  ● 데몬  ● 브릿지  [전체 시작] [전체 종료]
```

## 3. 모니터링 대상 누락 금지

> 시스템에 존재하는 프로세스는 전부 대시보드에 보여라.

- 구현 전에 "이 시스템이 동작하려면 뭐가 켜져 있어야 하지?" 질문
- 전제 조건(prerequisite)도 모니터링 대상
- 체크리스트: daemon, bridge, worker, DB, 외부 서비스 전부 나열

## 4. 이상 상태에 원인을 설명하라

> 숫자만 보여주지 말고 "왜"를 알려줘라.

- 비정상 값(0이 아닌 미읽음, offline 등)에 힌트 텍스트 추가
- 예: "MsgBus 30 미읽음" → "브릿지 offline → 전송 대기 중"
- 사용자가 다음 행동을 알 수 있어야 함

```python
# BAD
stats_card("MsgBus", f"{count} 미읽음")

# GOOD
hint = " (브릿지 offline)" if not bridge_alive and count > 0 else ""
stats_card("MsgBus", f"{count} 미읽음{hint}")
```

## 5. 중간 상태를 표현하라

> "시작 중", "종료 중" 같은 전이 상태도 보여줘라.

- alive/dead 이분법 → starting/running/stopping/offline 4단계
- 버튼 클릭 후 즉시 상태 변하지 않으면 spinner 또는 "시작 중..." 표시
- 사용자가 "내가 누른 게 먹힌 건가?" 의심하지 않게

## 6. 사용자 시점 자가 검증

> 만들고 나서 "상엽이 이걸 보면 바로 이해하겠나?" 자문하라.

구현 완료 후 체크:
- [ ] 다크모드에서 모든 텍스트가 읽히는가?
- [ ] 각 숫자/상태가 무엇을 의미하는지 라벨만으로 알 수 있는가?
- [ ] 이상 상태일 때 사용자가 다음 행동을 알 수 있는가?
- [ ] 항상 함께 동작하는 것이 분리되어 있지 않은가?
- [ ] 시스템 전제 조건이 전부 모니터링되고 있는가?
